Task 2 — Persistent Memory

Checkpoints · session IDs & follow-ups · two-tier memory · a durable per-user profile stored separately from chat history

The Problem

Task 1's agent is stateless: every query starts from a blank slate. The moment the process exits, everything it "knew" is gone. Task 2 makes the agent remember — across turns and across restarts.

Two distinct things must persist

  • The conversation itself (so follow-up questions like "and how many of those are refunds?" have context), and
  • durable facts about the user (so a stated preference like "I prefer concise answers with examples" shapes future answers — even in a brand-new conversation).

These have different lifetimes and different shapes, so the design uses two separate memory systems, deliberately kept apart.

What a Checkpoint Is

A checkpoint is a saved snapshot of the LangGraph state at a point in the graph's execution. Because our state holds the full message list, a checkpoint effectively saves the whole conversation so it can be reloaded later.

How LangGraph state becomes a checkpoint

The graph state carries the conversation under an add_messages reducer:

class AgentState(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]   # append-only history
    query_type: QueryType
    iteration_count: int
    session_id: str
    user_profile: dict[str, Any]
    pending_recommendation: str | None

The add_messages reducer means each step appends to the history rather than replacing it. A checkpointer writes this state to durable storage after each super-step, and reloads it when the same thread runs again.

How it's implemented — SQLite checkpointer

We use LangGraph's SqliteSaver, backed by a single database file:

# memory.py
def get_checkpointer(db_path="memory/checkpoints.db"):
    sqlite_module = import_module("langgraph.checkpoint.sqlite")
    SqliteSaver = getattr(sqlite_module, "SqliteSaver")
    conn = sqlite3.connect(db_file, check_same_thread=False)
    return SqliteSaver(conn)

# agent.py — graph is compiled WITH the checkpointer
graph = build_graph(checkpointer=get_checkpointer())
  • Storage: memory/checkpoints.db — a plain SQLite file, so state survives process exit with zero external services.
  • Keyed by thread: every checkpoint row carries a thread_id (our session ID — see next screen).
  • Graceful import: if the SQLite checkpointer package is absent, build_graph() falls back to an in-memory graph so the app still runs.
Why SQLite (not just an in-memory saver): it's persistent, file-based, requires no server, and is queryable — which is exactly how list_sessions() later enumerates known threads via SELECT DISTINCT thread_id FROM checkpoints.

Session IDs & Follow-up Questions

The session_id is the single thread key that ties everything together — it is the checkpoint's thread_id and the filename for the profile store. One ID, end to end.

How a session drives the checkpointer

# agent.py — run_query()
config = {"configurable": {"thread_id": session_id}}
for event in compiled_graph.stream(state, stream_mode="values", config=config):
    ...

Passing thread_id = session_id tells the checkpointer which conversation to load before the run and which to update after. Run again with the same ID and the prior message history is restored automatically — even after a full restart.

How follow-up questions work

Turn 1: "How many refund requests?"
↓   checkpoint saved under thread_id=alice
checkpoints.db
↓   (restart the app — state still on disk)
Turn 2 (same session): "And show me 3 examples of those"
↓   checkpointer reloads Turn 1 history first
agent sees full prior context → resolves "those"

Because the restored messages list is prepended to the new turn, the agent has the earlier exchange in context and can resolve references like "those" / "that category" without the user repeating themselves.

Session isolation. --session alice and --session bob use different thread_ids, so their histories never bleed into each other.
Discoverability. list_sessions() reads distinct thread_ids straight from the checkpoint DB, powering --list-sessions and the interactive sessions command.
One subtlety handled in Task 1's loop: because old turns are restored, the agent scopes its "do I have enough evidence yet?" checks to only the current user turn (_messages_since_latest_human) so a restored history can't trigger a premature final answer.

What "Memory" Is Here — and Why Two Kinds

"Memory" is split into two complementary systems with different shapes, lifetimes, and storage. This separation is the core design decision of Task 2.

episodic memory

What: the literal conversation — every human/AI/tool message of a session.

How: LangGraph SQLite checkpoints, keyed by thread_id.

Store: memory/checkpoints.db

Shape: append-only message log (verbatim, ordered).

Lifetime: the life of that one conversation thread.

Answers: "what did we just say?" → follow-up context.

profile memory

What: durable, distilled facts about the user.

How: LLM extraction → conservative merge → JSON file.

Store: memory/profiles/<session>.json

Shape: small structured dict (preferences, constraints, …).

Lifetime: grows and persists across all conversations.

Answers: "what kind of user is this?" → personalization.

Why not store everything in one place?

Different signal-to-noise. Raw chat history is verbose and full of one-off wording; a profile is a tiny set of durable facts. Mixing them would bloat the prompt and bury the signal.
Different access pattern. Episodic memory is replayed wholesale by the graph; the profile is injected as a compact summary into the system prompt — only the distilled facts, never the whole transcript.
Right tool per job. SQLite checkpoints are the LangGraph-native way to persist graph state; human-readable JSON is ideal for a small, inspectable, editable profile.
Robustness. If profile extraction fails, the conversation is unaffected, and vice versa — the two systems fail independently.
A third, related store handles pending query recommendations (memory/pending_recommendations/<session>.json) — same JSON + atomic-write pattern, used to remember a suggested-but-not-yet-confirmed query between turns.

Building & Maintaining a Persistent User Profile

After each answer, the agent distills durable facts from the latest exchange and merges them into a per-user JSON file — kept entirely separate from the conversation history.

The lifecycle (every turn)

1 · load profile JSON at query startload_user_profile(session_id)
↓ injected as a compact summary into the agent system prompt
2 · agent answers, informed by stored facts
↓ after the final answer
3 · extract durable facts from this exchangeupdate_user_profile(...) — small LLM
↓ conservative merge + dedupe
4 · atomic save back to <session>.json

1 · Extraction — turn chat into structured facts

A small, fast model (role profile) is given only the latest user+assistant snippet and a strict prompt: return JSON, only durable facts, ignore one-off requests, return {} if nothing qualifies. The stable schema: 

{
  "preferences":         [],   // e.g. "concise answers with examples"
  "frequent_categories": [],
  "frequent_intents":    [],
  "constraints":         [],
  "notes":               []
}

Using the cheap model here keeps this background step low-cost and low-latency.

2 · Conservative merge — never clobber, never bloat

_merge_profile() appends new list items, deduplicates (sorted(set(...))), skips empty values, and trims — so the profile grows monotonically with real facts and never fills with noise or duplicates.

3 · Durable, separate storage

  • One file per user: memory/profiles/<session_id>.json — physically separate from checkpoints.db.
  • Atomic writes: write to a temp file, then replace() — a crash mid-write can never corrupt the profile.
  • Safe IDs: session IDs are sanitized for filenames.
  • Fail-safe reads: missing or malformed JSON returns a clean default schema instead of crashing.
  • Never breaks a query: if extraction throws, the original profile is returned — a bad profile step can't fail the user's actual question.

4 · Using it — answering "what do you remember about me?"

The profile is injected into the system prompt via format_profile_for_prompt() on every turn, so it quietly shapes answers. And a direct memory question short-circuits to read straight from the stored profile: 

# agent.py
def is_memory_question(query):  # "what do you remember about me", ...
    ...
if is_memory_question(effective_query):
    answer = format_profile_for_prompt(profile)   # answered from the JSON store
Why this satisfies "stored separately": conversation history lives in SQLite checkpoints; the profile lives in its own JSON file. They're written by different code paths, have different schemas, and can be inspected/edited independently — yet are linked by the one shared session_id.

Run & Verify

Session controls (CLI)

python main.py --session alice                  # run within a named session
python main.py --list-sessions                  # list persisted sessions
# interactive commands:  profile  ·  sessions  ·  recommend  ·  exit

Verify episodic memory (follow-up across restart)

python main.py --session alice --query "How many refund requests did we get?"
# ...restart...
python main.py --session alice --query "Show me 3 examples of those"
# expect: the agent resolves "those" from the restored conversation

Verify the persistent profile

python main.py --session memory-smoke --query "I prefer concise answers with examples."
python main.py --session memory-smoke --query "What do you remember about me?"
# expect: the stored preference is returned from memory/profiles/memory-smoke.json

Verify isolation

python main.py --session alice --query "..."
python main.py --session bob   --query "What do you remember about me?"
# expect: bob's profile does NOT contain alice's facts
