Task 3 — FastMCP Server

Exposing the dataset analysis tools to any MCP client over streamable-HTTP — stateless and concurrency-safe

The Problem

Tasks 1–2 built a capable analyst, but its tools are locked inside our Python process. Task 3 publishes a subset of those tools over the Model Context Protocol (MCP) so that any MCP-aware client — Claude Desktop, another agent, a custom script — can call them remotely.

What Task 3 asks for

  • A FastMCP server that runs successfully.
  • At least 3 tools exposed over MCP.
  • Tools must be stateless and safe for concurrent clients.
  • README with startup + a client invocation example.

The core design tension

Our tools.py tools share a mutable working set (_STATE["filtered_df"]) so the agent can chain filter → count. That's great for one in-process agent turn, but fatal for an MCP server where many clients call concurrently — one client's filter would corrupt another's results. Task 3's real work is re-exposing this functionality without shared state.
The whole server is a single file: mcp_server.py.

What Is an MCP Server?

MCP (Model Context Protocol) is an open, standardized protocol for connecting LLM applications to external capabilities — tools, data, and prompts. Think of it as a universal adapter: instead of every model integrating with every tool in a bespoke way, both speak MCP.

The two roles

  • MCP server — exposes capabilities (our case: dataset analysis tools).
  • MCP client — an LLM app/agent that discovers and calls them.

What a server can expose

  • Tools — callable functions (what we expose).
  • Resources — readable data/context.
  • Prompts — reusable prompt templates.

Why this matters: the client discovers tools at runtime from the schemas the server advertises — it doesn't need to be coded against our specific functions ahead of time. The function name, docstring, and type hints become the public contract.

Transport

MCP runs over a transport. Common choices are stdio (local subprocess) and streamable-HTTP (networked). We use streamable-HTTP so the server is reachable as a normal network endpoint at http://localhost:8000/mcp, supporting remote and concurrent clients.

Creating a Server with FastMCP

FastMCP is a lightweight Python framework that turns plain functions into MCP tools with a decorator. It handles the protocol, schema generation, and transport — you write ordinary Python. Building the server is three small steps.

  1. Instantiate the server — give it a name and client-facing instructions. 
    from fastmcp import FastMCP

mcp = FastMCP(
    name="customer-service-analyst",
    instructions=(
        "Use these tools to analyze the Bitext customer service dataset. "
        "All exposed tools are stateless and safe for concurrent clients."
    ),
)
  2. Register tools with @mcp.tool() — the decorator inspects the function's name, type hints, and docstring and auto-generates the MCP tool schema the client will see. No manual schema writing. 
    @mcp.tool()
def list_categories() -> str:
    """List all dataset categories with row counts."""
    return list_categories_impl()

@mcp.tool()
def get_intent_distribution(category: Optional[str] = None) -> str:
    """Return intent distribution (optionally for one category)."""
    return get_intent_distribution_impl(category=category)
  3. Run it on a transport — one call starts the server. 
    if __name__ == "__main__":
    mcp.run(transport="streamable-http", host="0.0.0.0", port=8000)
That's the whole pattern. The docstring becomes the tool description the model reads to decide when to call it; the type hints (category: Optional[str] = None) become the input schema and defaults. Good names + docstrings + hints = a good MCP tool.

Architecture choice: thin MCP layer over existing logic

The MCP tools don't re-implement analysis. Where a function is already pure, the decorated tool is a one-line passthrough to the tools.py implementation (imported as ..._impl). Only where the original relied on shared state do we add a stateless wrapper (Screen 5). This keeps a single source of truth for the analysis logic.

Why Stateless Wrappers Are Essential

The single most important design rule of Task 3: MCP tools must not share mutable state. Here's the concrete reason and how it's solved.

The hazard

In tools.py, filter_by_category() writes to a module-level _STATE["filtered_df"], and count_rows() / show_examples() read it. For a single agent turn that's a feature. But an MCP server handles concurrent clients: 

Client A → filter REFUND   ┐
                           ├─ shared _STATE  ← race condition
Client B → filter SHIPPING ┘
Client A → count_rows()    → gets SHIPPING's count.  WRONG.

Shared in-process state means one caller can silently corrupt another's results.

The fix

Passthrough where already pure. list_categories, get_intent_distribution, and summarize_category compute from the full dataset every call — reused directly.
Wrap where state was used. show_examples and count_rows_for_intent are reimplemented to apply filters to a fresh load_dataset() view each call — never touching _STATE.
Force stateless flags. search_responses is always called with use_working_set=False, guaranteeing full-dataset behavior.
Deterministic results. Sampling uses a fixed random_state=42, so identical calls return identical output — predictable for clients and tests.
Net result: every MCP tool is a pure function of its arguments. Any number of clients can call in any order with no interference — exactly the concurrency-safety Task 3 demands. The in-process agent (Task 1) keeps its stateful chaining; the two worlds stay decoupled.

Connect a client (call one tool)

import asyncio
from mcp import ClientSession
from mcp.client.streamable_http import streamablehttp_client

async def main():
    async with streamablehttp_client("http://localhost:8000/mcp") as (r, w, _):
        async with ClientSession(r, w) as session:
            await session.initialize()
            result = await session.call_tool("list_categories", {})
            print(result.content[0].text)

asyncio.run(main())
