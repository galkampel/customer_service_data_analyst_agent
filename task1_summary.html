Task 1 — Customer Service Data Analyst Agent

Router · ReAct tool loop · transparent reasoning trace · max-iteration guard — built on LangGraph

The Problem

Build a conversational data analyst agent over the Bitext Customer Service dataset (≈26,872 rows of customer instructions, agent responses, categories, and intents). A user should be able to ask natural-language questions and get accurate, data-grounded answers.

What makes this non-trivial

  • Mixed query types. Some questions are precise and computational ("How many refund requests did we get?"); others are open-ended ("Summarize the FEEDBACK category"); some are entirely off-topic ("Who won the 2024 Champions League?"). Each needs different handling.
  • Answers must be grounded in data, not invented. A language model alone will happily fabricate counts and percentages. The agent must compute real numbers from the dataset.
  • Some answers require multiple steps. "How many cancel_order requests?" means first filter the data to that intent, then count — a chain, not a single lookup.
  • Reasoning must be transparent. The grader needs to see which tools were chosen, with what arguments, and what each returned.
  • It must terminate safely. A reasoning loop can spin forever; the agent needs a guaranteed stop.

The Solution & Why This Flow

A LangGraph state machine with a dedicated router in front of a ReAct agent/tool loop. The router decides the route; the agent reasons and calls typed tools; the loop continues until the agent writes a final answer or hits a guard. Source: agent.py.

Why a router-first design

Separation of concerns. Classification is a small, cheap decision. Pushing it into a dedicated node (on a smaller, faster model) keeps the expensive reasoning model focused only on questions that are actually in scope.
Safety / scope control. Out-of-scope queries are caught up front and refused politely — they never reach the tool loop, so the agent can't be coaxed into answering from general knowledge.
Right behavior per type. Structured vs. unstructured questions need different stopping rules (see Screen 4). Knowing the type early lets the agent node apply the correct policy.
Defensive defaults. The router output is model text, so it's parsed defensively (normalize_query_type) and, when uncertain, defaults to structured — preferring to try over wrongly refusing.

Why a ReAct loop (agent ⇄ tools)

The agent node binds the tools to the model. On each pass it either (a) emits one or more tool calls → control goes to the tools node, which runs them and feeds observations back, or (b) emits a final answer → the graph ends. This is the classic Reason → Act → Observe cycle, and it is what enables multi-step chaining: the model sees each observation and decides the next action.

Model split (why two models). The main reasoning/tool loop runs on the strong Llama-3.3-70B-Instruct; the router runs on the fast Llama-3.1-8B-fast. High-quality reasoning where it matters, low latency/cost on the cheap classification step.

The Tools — and Why Each Is Essential

Nine tools in tools.py, each a StructuredTool with an explicit name, an unambiguous description, and a Pydantic input schema. The schemas matter: they validate arguments (e.g. 1 ≤ n ≤ 50) and give the model a precise contract, which sharply improves tool-selection accuracy.

Several tools share a stateful "working set" (_STATE["filtered_df"]). A filter_* tool narrows the data; later tools operate on that narrowed set. This is the mechanism that makes multi-step chains both possible and efficient (Screen 4).

Discovery — "what is in the data?"

ToolDoesWhy essential
list_categories
()		 All categories with row counts. The entry point for an analyst who doesn't yet know the schema. Grounds every later question in real category names.
list_intents
(category)		 Intents inside one category, with counts. Bridges category → intent. Lets the model learn valid intent labels before filtering by them, avoiding guessed/invalid intents.

Filtering — set up the working set

ToolDoesWhy essential
filter_by_intent
(intent)		 Narrows the working set to one intent. Core of multi-step reasoning. Enables filter_by_intent → count_rows and similar chains. Returns a helpful error + sample intents if not found.
filter_by_category
(category)		 Narrows the working set to one category. Same role at the category level; sets up cheap follow-ups like count_rows or show_examples without re-scanning the full dataset.

Measurement — exact numbers

ToolDoesWhy essential
count_rows
()		 Counts the current working set (or full dataset). Guarantees grounded counts. The model never has to invent a number — counts are computed in pandas. Pairs with the filters to answer "how many X?"
get_intent_distribution
(category?)		 Intent counts and percentages, optionally per category. Answers "distribution / breakdown" questions in a single call, so the model isn't forced to chain many counts and do arithmetic itself.

Evidence — see the actual text

ToolDoesWhy essential
show_examples
(n, category?, intent?)		 Sample rows (customer + agent text). Qualitative questions ("show me examples") and grounding for summaries. Deterministic sampling (random_state=42) keeps runs reproducible.
search_responses
(keyword, n, use_working_set)		 Literal keyword search over instruction + response text. Bridges the gap when the user's wording doesn't map to an exact intent name. Literal (regex-off) matching is predictable and cheap; can search the filtered set for fast follow-ups.
summarize_category
(category, n_examples)		 One-shot summary: stats + intent breakdown + representative examples. Makes unstructured queries one-step. Bundles everything a summary needs into a single observation, so the agent gathers evidence once and then writes prose (no looping — see Screen 4).
Why typed schemas (Pydantic) instead of free-form args: they constrain ranges, mark optional vs. required fields, and document each argument inline. The model gets a precise contract → fewer malformed calls, and bad inputs fail fast with a clear message instead of a crash or a wrong answer.

Approach to Multi-step Reasoning

Multi-step reasoning is driven by the ReAct loop plus the shared working set, and kept safe by three layers of guards. The goal: chain when needed, stop as soon as the answer is in hand, and always terminate.

1 · How chaining actually happens

Example: "How many cancel_order requests are there?"

agent  → tool_call: filter_by_intent(intent="cancel_order")
tools  → observation: "Filter applied: intent='cancel_order' (category=CANCEL). Rows: 2,034."
agent  → tool_call: count_rows()
tools  → observation: "Current filtered working set: 2,034 rows (out of 26,872 total)."
agent  → final answer: "There are 2,034 cancel_order requests."

The filter_* tool writes to _STATE["filtered_df"]; count_rows reads it. The model only has to sequence the calls — state passing is handled by the tools, not crammed into the prompt. Tool descriptions even hint at the canonical chain ("Typical chain: filter_by_intent, then count_rows").

2 · Knowing when to stop (not just when to act)

The hard part of multi-step reasoning is avoiding needless extra steps. Three mechanisms:

Prompt discipline. The agent system prompt says: never call the same tool with the same args twice; one or two calls are usually enough; for summaries a single summarize_category call suffices.
Unstructured early-finalize. For unstructured queries, once one tool observation exists in the current turn, the agent is switched to a tool-less model and told to write the summary now — preventing summary loops.
Repeat-call breaker. _last_tool_call_repeats() detects two identical consecutive tool calls and forces a final answer with the tool-less model, so a stuck model can't spin on the same call.
Current-turn scoping. _messages_since_latest_human() ensures these decisions look only at the current user turn — old observations from a restored session can't trigger premature finalization.

3 · Guaranteed termination — the max-iteration fallback

Every pass through the agent node increments iteration_count. When it reaches MAX_ITERATIONS = 12, the agent stops looping and returns a graceful message instead of continuing: 

"I reached the maximum reasoning steps before completing this answer.
 Please rephrase the request or split it into smaller questions."

This is the hard backstop: no matter what the model does, the graph terminates.

Net effect: the agent chains tools when a question genuinely needs it (filter → count), finalizes as soon as the evidence is sufficient (especially for summaries), and can never run away — satisfying both the "multi-step reasoning" and "max-iteration fallback" requirements of Task 1.

Run & Debug

Reasoning visibility is built into both the CLI and LangGraph Studio.

CLI — reasoning trace printed inline

source .venv/bin/activate

# single query
python main.py --query "How many refund requests did we get?"

# interactive (type queries; see tool calls + observations before each answer)
python main.py

run_query() streams graph events and prints [tool call] …, [observation from …], and finally [final answer] — exactly the transparency Task 1 asks for.

LangGraph Studio — visual debug

langgraph dev

Open the printed URL, load the customer_service_agent graph, then:

  • Send a structured query and watch node order: START → router → agent → tools → agent → END.
  • Open the router output to confirm query_type.
  • Open agent/tool events to inspect the selected tool, its arguments, and the returned observation.
  • Send an out-of-scope query and confirm it routes to the refusal path with no knowledge answer.

python -c "import agent; print('graph initialized:', agent.graph is not None)"
