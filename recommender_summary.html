Query Recommender

Suggest → Refine → Confirm → Execute — a consent-gated "what should I ask next?" workflow

The Problem

A data analyst doesn't always know the next good question. The recommender helps the user decide what to ask next — but with one hard rule: it must never run a suggested query until the user explicitly confirms.

The four-step contract

  1. User asks for a recommendation.
  2. Agent suggests one concrete dataset query — and stops.
  3. User may refine, decline, or confirm.
  4. Agent runs the query only after clear confirmation.

Why this is more than "just call the LLM"

Consent gating. Suggesting and refining must call zero dataset tools — no side effects until "yes".
Cross-turn state. The pending suggestion must survive between separate CLI invocations / process restarts.
Robust yes/no detection. "yes" should confirm, but "yesterday" must not; "no" declines, but "knowledge" must not.
No stale execution. A declined suggestion must not be revived by a later "yes".
Logic lives in recommender.py (pure orchestration — no dataset tools), wired into the graph + cross-turn handling in agent.py, with the pending state persisted via memory.py.

Generating & Refining Suggestions

Both use the small recommender-role model at temperature=0.3 — cheap, fast, and low-risk because the output is just one short query string, never an executed action.

generate_suggestion(messages, user_profile)

Builds a context-aware prompt from three inputs and asks for exactly one query:

  • Valid categories — the 11 dataset categories, so suggestions stay executable.
  • User profile — via format_profile_for_prompt(), so suggestions reflect known preferences (Task 2 tie-in).
  • Recent history — last 6 messages, compacted, so the suggestion builds on the conversation.

The prompt insists on "only the suggested query text — no markdown, preamble, or explanation," and _clean_suggestion() strips stray quotes/whitespace.

refine_suggestion(pending, refinement)

Takes the current pending query + the user's change request and returns one revised query. The suggestion stays pending — refinement never executes anything.

pending"Compare top intents in REFUND and PAYMENT…"
user"make it about delivery instead"
refined (still pending)"Compare the top intents in DELIVERY and SHIPPING…"
Deterministic fallback. If the LLM/API is unavailable, both functions degrade gracefully: generation returns a fixed fallback query ("What is the distribution of intents in the REFUND category?") and refinement returns the unchanged pending query — so the CLI stays usable even with no API key.

Detecting Confirmation & Declination

Consent detection is rule-based, not an LLM call — it must be fast, deterministic, and free of false positives. This is the gate that decides whether real dataset tools run.

Why rules instead of a model

Determinism. "yes" must always mean yes — no temperature, no ambiguity on the step that triggers execution.
Zero latency/cost. A token-set check is instant; no extra API round-trip on every turn.

How it avoids false positives

Matching uses word-boundary tokenization (\b\w+\b), not naive substring search — so an embedded "yes" inside "yesterday" can't trigger confirmation: 

def is_confirmation(text):
    tokens = _word_tokens(text)                       # {"yes","run","it"}
    return bool(tokens & CONFIRM_KEYWORDS) or _contains_phrase(text, CONFIRM_PHRASES)

Keyword sets (yes/yeah/sure/ok/proceed…, no/nope/cancel/skip/stop…) plus phrase lists ("do it", "run it", "forget it", "not now") cover natural replies without overreaching.

Cross-Turn Persistence & Safety Guards

Because each CLI call builds a fresh input state, the pending suggestion is stored outside the graph, in a small JSON file — mirroring the Task 2 profile store.

Where the pending suggestion lives

save_pending_recommendation() / load_pending_recommendation() in memory.py persist to memory/pending_recommendations/<session>.json using the same atomic temp-file-then-replace write. Saving None deletes the file (clears pending).

How run_query() dispatches each turn

pending = load_pending_recommendation(session_id)
if pending:
    if is_confirmation(query):
        effective_query = pending            # swap in the suggested query
        save_pending_recommendation(session_id, None)   # clear first
        # → falls through to the normal agent/tool loop
    elif is_declination(query):
        save_pending_recommendation(session_id, None)   # clear, no tools
        return "Recommendation cancelled. Ask me anything else.", []
    else:
        refined = refine_suggestion(pending, query)     # treat as refinement
        save_pending_recommendation(session_id, refined)
        return format_suggestion_response(refined), []   # still pending, no tools

Safety guarantees

No auto-execution. Suggest & refine return early with empty step lists — the tool loop is never entered.
No stale revival. Decline clears the file immediately, so a later "yes" finds no pending state and does nothing stale.
Confirm-then-clear ordering. Pending is cleared before execution, so a crash mid-run can't replay the suggestion.
No profile noise. Suggest/refine/decline turns skip the profile update, so facts like "user said yes" never pollute the Task 2 profile.

Run & Verify

Full suggest → refine → confirm

python main.py --session demo --query "What should I query next?"
python main.py --session demo --query "make it about refund examples"
python main.py --session demo --query "yes, run it"
# 1st: suggestion, no tools · 2nd: refined, no tools · 3rd: runs the query (reasoning trace)

Decline clears pending (no stale execution)

python main.py --session cancel --query "What should I query next?"
python main.py --session cancel --query "no thanks"
python main.py --session cancel --query "yes"
# "no thanks" cancels; the later "yes" does NOT run the old suggestion
